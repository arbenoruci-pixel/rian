PATCH: ARKA worker cash handoff submit must use DB RPC only

Root cause fixed:
- The worker/UI handoff path could fall into the legacy/manual fallback path when explicit payment_ids were supplied.
- That fallback inserts cash_handoffs before inserting cash_handoff_items, so a later failure can leave a visible CANCELLED handoff shell with 0 items while payments remain open.

Changes:
- submitWorkerCashToDispatch now sends real READY_FOR_HANDOFF paymentIds and forces the atomic RPC path with rpcOnly/forceRpc.
- runArkaTransaction serializes same worker/payment handoff submits in-process to reduce double-submit races.
- submitHandoff now keeps any explicit payment_ids on submit_cash_handoff_atomic and does not route them through the manual fallback.
- Server verifies after RPC:
  - cash_handoff exists
  - status is PENDING_DISPATCH_APPROVAL
  - cash_handoff_items is non-empty and count matches payment_ids
  - item sum matches handoff.amount
  - arka_pending_payments moved to PENDING_DISPATCH_APPROVAL
  - handoff_note contains the handoff id
- Frontend service verifies the returned handoff/items before showing success.
- Worker pages add immediate click locks so “Dorëzo te Dispatch” cannot submit twice.

Expected Tapin/T347 result:
- gross 5.22
- m2 2.9 × 0.50 = 1.45 commission
- net handoff item/amount = 3.77
