AUDIT — PRANIMI / BAZ / “TË PA PLOTËSUARA” — DEEP SCAN V4

Scope inspected:
- app/pranimi/page.jsx
- lib/baseCodes.js only as direct code pool helper
- search across app/pastrimi/page.jsx, app/gati/page.jsx, lib/offlineStore.js, lib/syncManager.js, lib/syncEngine.js for draft/offline/edit bridge sources

Main finding:
PRANIMI does not have one single draft source. It has several parallel sources that can all influence the “TË PA PLOTËSUARA” UI or code lifecycle:

1) Local incomplete draft mirror
   - Keys: draft_orders_v1 and draft_order_<id>
   - Helpers: loadDraftIds(), rebuildDraftIdsByScan(), upsertDraftLocal(), removeDraftLocal(), readAllDraftsLocal(), readLocalDraftSummaries()
   - Used for real local/offline incomplete drafts.

2) Remote incomplete draft mirror in Supabase Storage
   - Bucket/path: tepiha-photos/drafts/*.json
   - Helpers: upsertDraftRemote(), deleteDraftRemote(), deleteDraftRemoteMany(), listDraftsRemote(), readDraftRemote(), fetchRemoteDraftsSummary()
   - Problem source: stale objects can remain even after public.orders exists.

3) Legacy numeric remote draft objects
   - Pattern: drafts/<number>.json
   - Live DB/Storage showed these often mean old shadow objects by order_id, not canonical draft ids.
   - Example: drafts/1306.json contains payload displayed as code 1306 / Artan bytyqi, but filename 1306 matches public.orders.id=1306 with status dorzim and a different order payload.
   - V2/V3 DB guard was right in principle, but this class needs a fail-closed local guard too because filename itself is legacy and unsafe.

4) Current PRANIMI session mirror
   - Key: tepiha_pranimi_current_session_v1
   - Helpers: readCurrentSessionLocal(), writeCurrentSessionLocal(), clearCurrentSessionLocal()
   - Used for active form restore, not meant to be a list source.

5) Code reservation lifecycle
   - Keys: base_order_code:<oid> and pranimi_draft_reservation:<local_oid>
   - Functions: tryReserveCodeInBackground(), markDraftReservationFromSnapshot(), persistMeaningfulDraft(), releaseBlankDraftReservation(), releaseExpiredBlankDraftReservations()
   - Helpers in lib/baseCodes.js: reserveSharedCode(), holdBaseCodeForDraft(), releaseLocksForCode(), markCodeUsed(), ensureUniqueBaseCodeForSave()

6) Offline/order outbox path
   - Functions: enqueueBaseOrder(), saveOrderLocal(), getAllOrdersLocal(), getPendingOps(), markPranimiLocalMirrorUnsynced()
   - This creates local orders/outbox rows, not UI drafts, but it can preserve code/local_oid state.

7) Edit bridges back into PRANIMI
   - Keys: tepiha_pastrim_edit_to_pranimi_v1, tepiha_gati_edit_to_pranimi_v1, tepiha_pranimi_active_edit_bridge_v1
   - Functions: consumePastrimiEditPayload(), hydrateFromPastrimiEdit(), clearActiveEditBridge()
   - These are edit flows for existing orders and must not be treated as new incomplete drafts.

Root cause after V3:
- The visible stale item 1306 was a legacy numeric remote object: drafts/1306.json.
- DB SQL confirmed the filename matches public.orders.id=1306 with status dorzim.
- Payload inside the stale Storage object can show a different client/code than the matching final order, so payload code/name alone is not safe.
- Therefore the UI needs a canonical rule: remote draft files named with only digits are legacy/unsafe and must not render as “TË PA PLOTËSUARA”. Real active drafts should use local_oid/UUID-style ids and local drafts can still show if they are meaningful.

V4 direction:
- Keep public.orders anti-join guard.
- Add a pre-DB fail-closed guard for remote numeric Storage filenames and REMOTE summaries missing their Storage file key.
- Add alias-based suppression so stale entries are removed by id, local_oid, file_key, remote_file_key, and legacy_order_id_candidate.
- Make HAP/FSHI pass the whole draft object so they know the original remote file key, not only draft.id.
