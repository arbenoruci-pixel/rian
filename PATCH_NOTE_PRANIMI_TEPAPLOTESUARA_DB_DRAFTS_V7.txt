PATCH NOTE — PRANIMI / TË PA PLOTËSUARA / DB-BACKED DRAFTS V7

Problem observed after V6:
- Stale Storage drafts were removed from the visible list, but a newly typed name followed by HOME still did not persist as a shared draft.
- Local draft safety was not enough because other workers cannot see localStorage.

Root cause:
- TË PA PLOTËSUARA still depended on localStorage / Storage behavior for incomplete drafts.
- The app needed a shared source-of-truth for incomplete drafts.

Fix:
- Meaningful PRANIMI drafts are now saved to public.orders with status='incomplete' and data.status='incomplete'.
- The DB row uses the same local_oid/draft_id that final save later uses.
- The shared TË PA PLOTËSUARA list now reads DB draft rows first via public.orders draft-like statuses.
- New Supabase Storage draft JSON objects are no longer written for PRANIMI incomplete drafts.
- Legacy Storage draft JSON is ignored for the visible list, preventing stale ghosts from returning.
- HOME waits for DB draft save when online. If meaningful work exists and DB draft save fails, HOME stays on PRANIMI and shows an error instead of silently losing the draft.
- Final save upserts the same public.orders row by local_oid and changes status to pastrim/gati flow as before.
- FSHI deletes only the matching draft-like DB row, then releases the code. It does not touch final/active orders.

Safety:
- No DB schema, RPC, or trigger changes.
- No Storage mass delete.
- No client row is created for a draft.
- No success/SMS/print behavior was loosened.
- Existing final orders are not edited.
