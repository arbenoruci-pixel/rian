PATCH NOTE - PRANIMI / TE PA PLOTESUARA - REMOTE STALE DRAFT FILTER V2

Problem:
Supabase Storage contained stale remote draft JSON files under tepiha-photos/drafts/*.json. Some of those drafts already had matching rows in public.orders with statuses such as pastrim, gati, dorzim, done, or delivery. The UI could still show them in TE PA PLOTESUARA.

Patch:
1. Added fail-safe draft/order matching before rendering TE PA PLOTESUARA.
   The list now does a bulk DB anti-join guard against public.orders before render. A draft is hidden if any matching public.orders row exists and that order status is not a draft/incomplete status.

2. Matching now checks:
   - draft.local_oid vs orders.local_oid
   - draft.local_oid vs orders.data.local_oid
   - draft.local_oid vs orders.data.pranimi_code_lifecycle.local_oid
   - Storage file key drafts/<id>.json vs orders.id for legacy files
   - draft.code vs orders.code

3. Remote Storage drafts are not trusted without a DB anti-join check.
   If DB verification is uncertain for a remote draft, the UI hides it instead of rendering it as incomplete.

4. HAP is blocked for stale drafts.
   If the user clicks HAP and the draft matches a final/active order, the form is not opened and no new code is assigned.

5. FSHI is blocked for stale drafts.
   If the draft matches a final/active order, the UI removes it from the local visible list but does not release the code and does not do mass remote cleanup.

6. After final order DB verify succeeds, remote draft cleanup now removes:
   - drafts/<local_oid>.json
   - drafts/<order_id>.json when a legacy draft object exists
   - related local/session draft IDs from the same verified save flow

Not changed:
- No DB schema changes
- No RPC changes
- No mass delete/update
- No transport/arka/pastrim/gati/dispatch changes
- No existing data cleanup

Important behavior:
Storage may still contain stale objects until a separate reviewed cleanup is done, but TE PA PLOTESUARA no longer renders them when public.orders proves the order exists.
